# Analytics & Reporting

## Overview

The analytics system provides two layers of insights:

1. **AI Insights** — AI-evaluated conversation KPI metrics (satisfaction, sentiment, quality, efficiency, resolution) generated by LLM analysis of call center transcripts.
2. **Agent Performance & Node Analytics** — Pre-aggregated execution metrics for AI agents and workflow nodes, summarized from `agent_response_logs` into daily stats tables.

**Data freshness**: Execution stats are updated in near real-time via background events on each execution, with a Celery worker running twice daily as a full-recount fallback. AI Insights metrics are computed on-the-fly from `conversation_analysis` records.

## Architecture

```
                          DATA INGESTION
                          ─────────────

  Chat request
       │
       ▼
  agent_response_log saved ──► asyncio.create_task()
       │                              │
       │                              ▼
       │                   analytics_realtime.py
       │                   INSERT ON CONFLICT += 1
       │                   (fire-and-forget, try/except)
       │
       ▼
  Celery (twice daily)
  analytics_aggregation.py
  Full recount from raw logs (batch size: 1000)
  Overwrites incremental values

```

## Database Tables

### `agent_execution_daily_stats`

One row per (agent, date). Unique constraint: `(agent_id, stat_date)`.

| Column | Type | Description |
|--------|------|-------------|
| `agent_id` | UUID | FK to `agents.id` |
| `stat_date` | date | The day these stats cover |
| `execution_count` | int | Total agent executions |
| `success_count` | int | Executions with status "success" or "completed" |
| `error_count` | int | Executions that errored |
| `avg_response_ms` | float | Average response time (ms) |
| `min_response_ms` | float | Fastest response (ms) |
| `max_response_ms` | float | Slowest response (ms) |
| `total_response_ms` | float | Sum of all response times (used for weighted avg) |
| `total_nodes_executed` | int | Total workflow nodes run across all executions |
| `avg_success_rate` | float | Average per-execution node success ratio |
| `total_success_rate_sum` | float | Sum of per-execution node success ratios (used for weighted avg) |
| `rag_used_count` | int | Executions that used RAG/knowledge base |
| `unique_conversations` | int | Distinct conversation IDs |
| `finalized_conversations` | int | Conversations marked finalized |
| `in_progress_conversations` | int | Conversations still in progress |
| `thumbs_up_count` | int | Positive feedback count |
| `thumbs_down_count` | int | Negative feedback count |
| `last_aggregated_at` | datetime | When this row was last updated |

### `node_execution_daily_stats`

One row per (agent, node_type, date). Unique constraint: `(agent_id, node_type, stat_date)`.

| Column | Type | Description |
|--------|------|-------------|
| `agent_id` | UUID | FK to `agents.id` |
| `node_type` | string | Workflow node type (e.g. `llm_node`, `condition_node`) |
| `stat_date` | date | The day these stats cover |
| `execution_count` | int | Times this node type was executed |
| `success_count` | int | Successful node executions |
| `failure_count` | int | Failed node executions |
| `avg_execution_ms` | float | Average node execution time (ms) |
| `min_execution_ms` | float | Fastest execution (ms) |
| `max_execution_ms` | float | Slowest execution (ms) |
| `total_execution_ms` | float | Sum of all execution times |

## Data Ingestion

### Real-time incremental updates

**File**: `backend/app/services/analytics_realtime.py`

When an agent response log is saved (`chat_as_client_use_case.py`), a background task fires
via `asyncio.create_task()`. It parses the agent response and runs atomic PostgreSQL upserts:

All fields are updated in near real-time via background tasks. The Celery worker serves only as a
full-recount fallback to correct any drift.

**On each agent execution** (`chat_as_client_use_case.py` → `update_stats_incrementally`):
- Counters: `execution_count`, `success_count`, `error_count`, `rag_used_count`, `total_nodes_executed`
- Timing: `avg/min/max/total_response_ms`, `avg/min/max/total_execution_ms`
- Node success rate: `avg_success_rate`, `total_success_rate_sum` (running weighted average)

**On conversation lifecycle** (`conversations.py` start/finalize endpoints):
- `unique_conversations` — incremented when a conversation starts
- `in_progress_conversations` — incremented on start, decremented on finalize
- `finalized_conversations` — incremented when a conversation is finalized

**On message feedback** (`conversations.py` add-feedback endpoint → `update_feedback_given`):
- `thumbs_up_count` — incremented for GOOD/VERY_GOOD feedback
- `thumbs_down_count` — incremented for BAD/VERY_BAD feedback

The finalize and feedback hooks look up `agent_id` via `conversation → operator → agent` in the background task.

**Error handling**: All background tasks are wrapped in `try/except`. If they fail,
a warning is logged but nothing propagates to the user. Celery corrects on its next run.

### Celery batch aggregation

Controlled by `CELERY_ENABLE_AGGREGATE_AGENT_ANALYTICS_TASK` env variable (default: `true`).

Runs twice daily (2 AM + 2 PM UTC). Fetches `agent_response_logs` since the last aggregation
(or last 30 days on first run) in batches of 1000 rows, parses each log's `raw_response` JSON,
groups by (agent, date) and (agent, node_type, date), and upserts complete rows. This is the
authoritative source of truth — it overwrites any incrementally-updated values with a full recount.

**Multi-tenancy**: The Celery task runs for master DB and all active tenants sequentially,
using `run_task_with_tenant_support()`.

## Frontend Pages

### AI Insights (`/analytics`)

- **Metric cards**: 2 rows of 4 — Responsiveness, Customer Satisfaction, Quality of Service, Resolution Rate, Efficiency, Positive Sentiment, Negative Sentiment, Analyzed count
- **Tooltips**: Each metric has an info tooltip describing what it measures
- **Performance chart**: Daily trend (satisfaction, quality, resolution) via area chart
- **Filters**: Agent selector, date range picker with presets (default: last 7 days)

### Agent Performance (`/analytics/agent-performance`)

- **Summary cards**: Conversations, total executions, success rate, avg response time, thumbs up/down
- **Area chart**: Daily success/error trends, stacked by agent when viewing all agents
- **Escalation tracking**: Select a node type as "escalation node" to compute escalation/containment rates (persisted in localStorage per agent)
- **Data table**: Daily stats (single agent) or aggregated per-agent stats (all agents)
- **Row click**: Opens node breakdown dialog with donut chart and detail table
- **Export button**: CSV / Excel / PDF
- **Filters**: Agent selector, date range picker with presets (default: last 7 days)

### Node Analytics (`/analytics/node-analytics`)

- **Bar chart**: Top 12 node types by execution count (horizontal bars: executions, success, failures)
- **Data table**: Aggregated stats by agent x node type, with weighted average execution time
- **Filters**: Agent selector, date range picker with presets (default: last 7 days), node type filter
- **Export button**: CSV / Excel / PDF

### Shared components

- **`AnalyticsFilters`**: Reusable filter bar with agent selector, date range picker, and slot for extra controls (children)
- **`DateRangePicker`**: Generic date range picker component with preset sidebar (Today, Last 7/30 days, This week/month, Last month/year, Year to date, Clear)
- **`ExportButton`**: Dropdown for CSV/Excel/PDF export with loading state
- **`useAgentsList`**: Shared hook for fetching the agent list across all analytics pages
- **`analyticsParams.ts`**: Shared utility for `toMetricsApiParams()` and `buildQueryString()` — used by hooks, charts, and services

## Key Calculations

| Metric | Formula | Notes |
|--------|---------|-------|
| Success rate | `success_count / execution_count * 100` | Per agent or per node type |
| Avg response time | `SUM(avg_ms * exec_count) / SUM(exec_count)` | Weighted average across days |
| Escalation rate | `escalation_node_executions / total_executions` | User-selected escalation node |
| Containment rate | `1 - escalation_rate` | Conversations handled without escalation |
| Node avg exec time | `total_execution_ms / execution_count` | Weighted by execution volume |
| AI KPI scores | `AVG(score) * 10` | LLM scores are 0-10, displayed as 0-100% |

**Why weighted averages?** A simple `AVG(daily_avg)` would give equal weight to a day with
1 execution and a day with 1000. Weighted averages account for volume differences.
